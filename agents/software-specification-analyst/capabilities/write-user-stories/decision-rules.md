# Decision Rules

- Escrever historias a partir de requisitos e objetivos confirmados.
- Usar formato "Como..., quero..., para..." quando `story_format` nao indicar outro padrao.
- Cada historia deve ter criterios de aceite verificaveis.
- Incluir Gherkin somente quando reduzir ambiguidade ou for solicitado.
- Relacionar historias a epicos, features, dependencias, riscos e perguntas abertas.
- Nao inventar ator, beneficio ou comportamento para completar template.
- Separar historia funcional de tarefa tecnica quando necessario.
- Marcar historia bloqueada quando requisito ou regra de negocio estiver pendente.
- Evitar historias grandes demais sem fatiamento claro.
- A saida deve alimentar rastreabilidade, QA e planejamento de desenvolvimento.
