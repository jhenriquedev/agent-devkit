# Decision Rules

- Construir matriz apenas a partir de requisitos, historias, criterios, componentes, testes e riscos identificados.
- Cada requisito funcional ou nao funcional deve ter identificador estavel.
- Relacionar historias e criterios de aceite a requisitos especificos, sem agrupamentos vagos.
- Marcar gaps quando requisito nao tiver historia, criterio, teste ou componente associado.
- Separar rastreabilidade confirmada de inferida.
- Nao inventar cobertura de teste ou componente tecnico ausente.
- Incluir perguntas abertas que bloqueiam implementacao ou QA.
- Preservar referencias de origem para auditoria.
- Manter matriz legivel em Markdown.
- A saida deve alimentar review de completude e handoff para desenvolvimento.
