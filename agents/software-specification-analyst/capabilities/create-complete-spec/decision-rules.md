# Decision Rules

- Consultar `vendor/skills/CATALOG.md` e carregar apenas skills relevantes ao dominio.
- Usar `product-capability` como base para explicitar contrato, restricoes e perguntas abertas.
- Separar fatos, inferencias, premissas e lacunas antes de redigir a spec.
- Nao inventar requisitos, regras de negocio, decisoes de produto ou restricoes tecnicas.
- Incluir todas as secoes obrigatorias da `specification_policy`.
- Gerar artefatos funcionais, tecnicos, historias, jornadas, rastreabilidade e perguntas abertas.
- Usar Mermaid para fluxos quando houver jornada.
- Se houver pergunta bloqueante, destacar a lacuna e nao marcar a spec como pronta para implementacao.
- Pedir confirmacao antes de criar pasta `specifications/<slug>` ou sobrescrever arquivos.
- Manter paths portaveis e nomes de artefato previsiveis.
