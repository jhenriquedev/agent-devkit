# Problemas Pendentes Da v0.3.0

Este documento consolida os pontos que ainda estao parciais ou abaixo do nivel
esperado para a visao completa do Agent DevKit como agente autonomo, local-first,
multiagente, extensivel e operavel via CLI/MCP.

## 1. Conversa Sem LLM Externa

Estado atual: parcial.

- O CLI responde fluxos deterministas, onboarding, status e algumas acoes locais.
- A conversa livre local depende de Ollama com `qwen3:0.6b` configurado.
- O pacote nao embarca os pesos do modelo; o setup precisa baixar/configurar o
  runtime local.

Resultado esperado:

- Permitir uma experiencia inicial conversacional local, mesmo antes de conectar
  Claude, Codex, OpenAI, Anthropic ou OpenRouter.
- Degradar com clareza quando o modelo local nao estiver disponivel.

## 2. Knowledge Remoto

Estado atual: parcial.

- Existem providers e manifests para Google Drive, Notion, GitHub, SharePoint e
  outros destinos.
- A knowledge base local file-first existe.
- O sync remoto ainda e planejado e retorna `remote_connected: false`.

Resultado esperado:

- Sincronizar de verdade com providers remotos suportados.
- Preservar ownership, curadoria, chaves de contribuicao e revisao antes de
  publicar conteudo na memoria principal.

## 3. Instalacao Completa De CLIs Necessarias

Estado atual: parcial.

- Existe toolchain para detectar e instalar CLIs como Claude, Codex, GitHub,
  Playwright, AWS, Ollama, Node, Python e Git.
- Algumas instalacoes sao executaveis com `--yes`; outras ainda sao instrucoes
  manuais por plataforma.
- O onboarding ainda nao e um fluxo universal polido que instala, valida e
  configura tudo de forma guiada.

Resultado esperado:

- Onboarding completo e minimo, ambos interativos.
- Instalacao/validacao idempotente das CLIs necessarias.
- Configuracao sob demanda quando uma capability exigir provider ou ferramenta.

## 4. Docker

Estado atual: parcial.

- O agente Docker analisa alvo, gera Dockerfile/compose, revisa seguranca e
  cria planos de build.
- Build, push e deploy reais permanecem bloqueados por padrao.

Resultado esperado:

- Manter seguranca por padrao, mas oferecer fluxo confirmado para build local e
  validacao real quando o usuario autorizar.

## 5. Agente Desenvolvedor Com Integracao GitHub

Estado atual: parcial.

- Existem GitHub provider, revisao de PRs e fluxo de contribuicao.
- Ainda nao ha um agente desenvolvedor completo e maduro para implementar
  features ponta a ponta com branch, patch, testes, PR e revisao final.

Resultado esperado:

- Usar Claude/Codex como coordenadores/revisores.
- Usar capacidades deterministicas e automacoes locais para tarefas repetitivas.
- Integrar com GitHub sem assumir escrita externa sem permissao.

## 6. Criacao De Novos Agentes E Automacoes Pelo Proprio Agent

Estado atual: parcial.

- O runtime cria scaffolds locais de agentes, scripts, skills e automacoes.
- Existem builders especializados para agentes genericos, agentes internos,
  Python, Selenium e PyAutoGUI.
- Ainda falta um fluxo autonomo completo que projete, implemente, teste,
  registre e evolua agents/capabilities complexas com qualidade consistente.

Resultado esperado:

- Criador de agentes capaz de transformar necessidade recorrente em agente,
  capability, automacao, knowledge e entrada de catalogo.
- Criador de automacoes capaz de promover rotinas repetitivas para scripts
  locais reutilizaveis sob `.agent-devkit`.

## 7. Memoria Do Modo De Trabalho Do Usuario

Estado atual: parcial.

- Existem memoria local, perfil, preferencias, projetos, rotinas, napkin,
  historico de uso e sessoes.
- A memoria ainda depende muito de dados explicitos e nao opera como memoria
  semantica profunda com curadoria automatica robusta.

Resultado esperado:

- O agente deve aprender preferencias estaveis, modo de trabalho, nomes,
  projetos recorrentes e decisoes persistentes.
- Mudancas sensiveis devem passar por curadoria e evitar armazenamento de
  segredos.

## 8. Gestao Completa De Sessoes

Estado atual: parcial.

- O runtime lista, mostra e retoma sessoes.
- Cria sessao ativa, nova sessao e aplica contexto recente ao prompt.
- Ainda faltam operacoes ricas como renomear, arquivar, excluir, buscar e
  organizar sessoes.

Resultado esperado:

- Sessoes devem ser uma superficie completa de trabalho continuado.
- O comando `agent` sem argumentos deve reconhecer conversa em andamento,
  pendencias, tarefas e contexto ativo.

## 9. Multiagente Com Claude/Codex Como Coordenadores

Estado atual: parcial.

- Existe planejamento multiagente, grafo de colaboracao, selecao de
  especialistas, review gate e roteamento de modelos.
- Claude/Codex podem atuar como backends coordenadores/revisores.
- Ainda nao e uma execucao colaborativa profunda com multiplos subagentes reais
  trabalhando em paralelo de forma robusta.

Resultado esperado:

- Claude/Codex devem coordenar, planejar, revisar e decidir delegacoes.
- LLM local deve executar tarefas operacionais e repetitivas quando suficiente.
- Subagentes/capabilities devem produzir handoffs auditaveis e revisaveis.

## 10. MCP Completo

Estado atual: parcial.

- Existe MCP stdio funcional com muitas ferramentas expostas.
- Algumas areas ainda dependem de `capability_run` generico.
- Sessoes nao aparecem como superficie MCP completa.
- Knowledge remoto ainda nao executa sync real.

Resultado esperado:

- MCP deve cobrir as superficies principais do produto: agentes, capabilities,
  sessoes, memoria, knowledge, tarefas, automacoes, toolchain, providers,
  configuracao, execucao e diagnostico.
- Deve ser util por Claude Desktop, Codex App, Claude CLI, Codex CLI e outros
  hosts compativeis.

## 11. Scheduler/Daemon De Tarefas Agendadas

Estado atual: parcial.

- Existem tarefas locais e `scheduler run-once`.
- O daemon continuo ainda nao e iniciado pelo CLI.
- A execucao recorrente depende de cron, systemd, launchd ou outro agendador
  externo.

Resultado esperado:

- Oferecer instalacao/configuracao guiada do agendador local.
- Manter notificacoes de inicio, falha, bloqueio, retry e conclusao.
- Preservar opt-in para tarefas com efeitos externos.
